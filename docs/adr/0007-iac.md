# ADR-0007: IaC Tool, Layout, and Performance Budgets

- **Status:** Accepted
- **Date:** 2026-05-08
- **Deciders:** Jason
- **Tags:** iac, terraform, opentofu, aws, performance

> Format: MADR 4.x (bundled sub-decisions form). See `template.md`.

## Context and Problem Statement

The platform supports AWS-deployed projects (a stated stack target) and a key portfolio piece is the SAP-C02 study work, which is AWS-architecture-heavy. IaC is the primary mechanism for declaring AWS infrastructure as a portfolio artifact rather than as console clicks.

The question: which IaC tool, repo layout, state backend, environment-separation pattern, drift/cost handling, and performance posture together make AWS infrastructure manageable for a solo developer with AI labor?

This ADR bundles six sub-decisions because the tool choice constrains layout, layout constrains state backend, state backend interacts with environment separation, and the performance posture is a cross-cutting requirement explicitly raised by the user during proposal review.

## Decision Drivers

- **SAP-C02 portfolio relevance.** The cert curriculum expects familiarity with CloudFormation/CDK; in industry, Terraform dominates the job-description signal. Showing Terraform competence is more valuable than showing CloudFormation competence even though the cert references the latter more.
- **AI labor pool.** LLMs are notably better at Terraform than at CDK or Pulumi (training corpus size). The bet on AI doing the lifting depends on tool familiarity to the AI.
- **Solo developer.** Tool/state setup must be quick. Heavy machinery like Atlantis or Spacelift is overkill.
- **Performance.** This came up explicitly during proposal review — IaC choices must not penalize commits that don't touch IaC. Path-filter discipline is required.
- **Live-data project (Game Night) needs real infra.** Decisions made now will be applied to actual infrastructure within weeks, not theoretically.
- **License risk.** HashiCorp's BSL license shift in 2023 introduced commercial-use restrictions on Terraform. Whether to stay on Terraform vs. move to OpenTofu is a real choice.

## Considered Options

The bundle has six sub-decisions:

- **Sub-decision 1 — IaC tool:** chose **OpenTofu** (CDK acceptable via project ADR)
- **Sub-decision 2 — Repository organization:** chose **Hybrid** (shared modules + per-project `terraform/`)
- **Sub-decision 3 — State backend:** chose **S3 + DynamoDB**
- **Sub-decision 4 — Environment separation:** chose **Directory-per-env**
- **Sub-decision 5 — Drift, cost handling:** chose **Weekly drift + Infracost on PR + AWS Budgets**
- **Sub-decision 6 — Performance posture:** chose **Explicit budgets** (added during proposal review at user's question)

## Decision Outcome

We adopt:

1. **OpenTofu** as the platform default IaC tool. CDK acceptable per-project via ADR for TS-only AWS-only projects with shared-language value.
2. **Hybrid repository organization**: shared modules in this platform repo (`templates/_shared/terraform-modules/`), versioned via Git tags; per-project `terraform/` directories consume them with pinned refs.
3. **S3 + DynamoDB state backend** with one-time per-account bootstrap script. State key pattern: `<project>/<env>/terraform.tfstate`.
4. **Directory-per-env** separation (`terraform/envs/dev/`, `staging/`, `prod/`) — not workspaces.
5. **Weekly scheduled drift detection** with `drift-detector` subagent triage.
6. **Infracost on PR** + **AWS Budgets** for cost management.
7. **Explicit performance budgets** as part of the standard, with path filters, plugin caching, and parallel jobs baked into reusable workflows.

## Consequences

### Positive

- SAP-C02-canonical AWS infra-as-code (Terraform + S3-backed state + IAM/OIDC) — directly studyable artifact.
- Shared modules give cross-project leverage (one VPC pattern, one IAM-OIDC pattern, one observability pattern).
- Directory-per-env makes `terraform plan` outputs honest — what you see is exactly what changes for that env.
- Drift detection prevents the "we have IaC but nothing matches" failure mode that plagues mature teams.
- Performance budgets prevent IaC discipline from becoming a tax on app-only commits.
- License risk eliminated by choosing OpenTofu.
- LLM reliability advantage (vs CDK / Pulumi) compounds with the AI labor model — agents write Terraform fluently.

### Negative

- **Two-step bootstrap.** S3 + DynamoDB must exist before Terraform runs against an account. Mitigated by the one-time `bootstrap-tfstate.sh` script.
- **Module versioning discipline required.** Sourcing from `main` is forbidden; a reckless ref change breaks downstream projects. Mitigated by release-please managing module tags.
- **Some duplication** in per-env `main.tf` files. Acceptable cost for clarity.
- **Drift-detector adds an agent to define.** Folded into ADR-0011 (AI workflows); the agent is in the 12-specialist roster.
- **Terratest costs real money** on each shared-module test run. Restricted to `templates/_shared/terraform-modules/` and run on tag, not on PR.

### Neutral

- We're committed to OpenTofu over Terraform proper. Future migration back to Terraform (if HashiCorp re-licenses) is mechanical.
- Choice of Terraform over CDK locks us out of IDE refactoring tools. Acceptable.
- AWS-first orientation. Multi-cloud is not a current requirement; if it becomes one, Terraform is the right tool for it (which is partly why we chose Terraform/OpenTofu).

## Pros and Cons of the Options

### Sub-decision 1: IaC tool

| Option | Trade-off |
|---|---|
| **Terraform (HashiCorp, BSL-licensed)** | Largest ecosystem; license concerns. |
| **OpenTofu** (chosen) | API-compatible fork; MPL-licensed; community has substantively migrated by 2026; same ecosystem. |
| **AWS CDK** | Real language (TS/Python); AWS-only; CloudFormation-under-the-hood quirks; smaller community; LLMs less reliable. |
| **Pulumi** | Real languages, multi-cloud; smaller community; state backend complexity. |
| **CloudFormation** | AWS-native; SAP-C02-relevant for cert; verbose; not a default in 2026. |
| **AWS SAM** | Niche (serverless-only); Terraform handles the same with broader scope. |

### Sub-decision 2: Repository organization

| Option | Trade-off |
|---|---|
| **Per-project `terraform/` only** | Loose coupling; easy onboarding; no shared modules; rederivation per project. |
| **Monorepo for all infra** | Maximum reuse; central state; access control gets awkward; deploys couple. |
| **Hybrid** (chosen) — shared modules in this platform repo + per-project `terraform/` | Reuse where it matters; per-project autonomy; needs versioning discipline on shared modules (mitigated by release-please + Git tags). |

### Sub-decision 3: State backend

| Option | Trade-off |
|---|---|
| **S3 + DynamoDB** (chosen) | Free at solo scale; AWS-native; SAP-C02-canonical; one-time bootstrap. |
| **Terraform Cloud / HCP Terraform** | Managed UI; remote runs; free tier limits; vendor lock-in. |
| **Local state** | Anti-pattern; no concurrent access; lost on machine swap. |

### Sub-decision 4: Environment separation

| Option | Trade-off |
|---|---|
| **Workspaces** | Single config; soft-discouraged by HashiCorp's own updated guidance for env separation; provider-difference gotchas. |
| **Directory-per-env** (chosen) | Clear separation; per-env state independence; some duplication mitigated by shared modules. |
| **Separate root modules per env** | Full isolation; high duplication; defeats module composition. |

### Sub-decision 5: Drift, cost handling

| Concern | Choice | Why |
|---|---|---|
| Drift detection | Weekly scheduled `plan` + `drift-detector` agent | Industry-standard cadence; agent triages so human doesn't see noise. |
| Cost preview | Infracost on PR (only when `terraform/` changed) | Catches surprise AWS bills before they hit; free for solo. |
| Budget alerts | AWS Budgets per project per env | Backstop for missed Infracost surprises. |

### Sub-decision 6: Performance posture (added during review)

The user pushed back during proposal review with: *"Commits shouldn't cost tens of minutes based on IaC choices."* This deserved an explicit response in the ADR rather than being treated as an implementation detail.

| Option | Trade-off |
|---|---|
| **Implicit / hopefully fast** | Standard, but offers nothing to push back against if regressions creep in. |
| **Explicit budgets baked into the standard** (chosen) | Testable claims; budget violations trigger architect-led investigation. |
| **Performance audited only on complaint** | Reactive; gaps compound silently. |

Budgets:

- App-only PR: 0s IaC overhead (path filters)
- IaC-touching PR: ≤3 min IaC overhead (parallel jobs, plugin caching)
- Pre-commit (any change): ≤10s total (per ADR-0005)
- Deploy: ≤3 min for solo-scale projects
- Drift detection: out-of-band (weekly schedule)

## Implementation notes

- Standards doc: [`docs/standards/08-iac.md`](../standards/08-iac.md).
- Bootstrap script: `scripts/bootstrap-tfstate.sh` (~30 lines AWS CLI; one-time per account).
- Shared module starter set: `templates/_shared/terraform-modules/{vpc, iam-github-oidc, lambda-base, ecs-service, observability-base}` — to be authored as part of Task #14 (per-stack templates).
- Reusable workflows: `workflows/terraform-pr.yml`, `terraform-deploy-{dev,staging,prod}.yml`, `drift-detect.yml`, `infracost.yml` — Task #15.
- Performance budget enforcement: each reusable workflow's runtime is logged in the digest; budget regressions become architect investigations.
- The `drift-detector` agent's role and triggers are defined in ADR-0011 (AI workflows).
