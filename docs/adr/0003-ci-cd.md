# ADR-0003: CI/CD Pipeline & Approval Model

- **Status:** Accepted
- **Date:** 2026-05-08
- **Deciders:** Jason
- **Tags:** ci-cd, deployment, ai-autonomy, approval-model

> Format: MADR 4.x (bundled sub-decisions form). See `template.md`.

## Context and Problem Statement

This decision sits at the heart of the platform's value proposition: who (or what) has authority to ship code to production? The tension is that the safer-on-paper pattern (human-approves-prod) conflicts with the operator's working profile that has to operate it. A gate that the operator skips under fatigue is worse than no gate at all.

The question: what approval model and pipeline shape gives a solo developer with live-data projects safe-enough shipping authority without forcing them into a per-deploy review loop?

This ADR bundles five sub-decisions because the approval model, environment topology, and promotion mechanism are all tightly coupled.

## Decision Drivers

- **Live data exists.** At least one project (Game Night) handles real user data. Bugs in prod aren't free; data corruption is irreversible.
- **The human's working profile.** Jason wants to operate in the Wonder + Discernment lane (brainstorming, design choices, architectural decisions) and explicitly *not* in the execution-review lane (PR review, deploy approval). His Working Genius profile is Wonder + Discernment; his Frustrations include Galvanizing and Enablement. His 8w7 + finisher-gap pattern means manual approval gates will silently rot under fatigue.
- **AI is the labor pool.** Modern AI tooling (subagents, hooks, GitHub Actions integrations) can handle the full PR review + deployment lifecycle. Question is whether it *should* be trusted, and where the boundaries are.
- **Irreversibility class.** Some changes (destructive migrations, security alterations) cannot be auto-rolled-back. They need a different category of gate.

## Considered Options

The bundle has five sub-decisions:

- **Sub-decision 1 — Approval model:** chose **AI shipping authority + ADR-gated human decisions**
- **Sub-decision 2 — Environment topology:** chose **dev+staging+prod for live-data; dev+prod for experiments** (project-configurable)
- **Sub-decision 3 — Promotion mechanism:** chose **Trigger-based throughout, tag-based to prod**
- **Sub-decision 4 — Deploy strategy:** chose **Per-stack defaults with automated rollback**
- **Sub-decision 5 — ADR-gated category list:** chose **5 categories** (destructive migrations, new external deps/services, security-relevant, API contract, schema/data-model)

## Decision Outcome

We adopt the **AI-shipping-authority** model:

1. **Two environment topologies**, project-configurable: `dev+staging+prod` for live-data projects (default), `dev+prod` for experiments.
2. **Fully automated trigger-based promotion** through environments. Tag-based to prod via release-please.
3. **AI is the deploy gate.** Multi-layer review (code-reviewer, security-reviewer, destructive-change detector, test pyramid, smoke tests, post-deploy health watch with auto-rollback).
4. **Five ADR-gated change categories** block auto-merge until a paired ADR exists and is Accepted: destructive DB migrations, new external deps/services, security-relevant changes, API contract changes, schema/data-model changes. The architect drafts the ADR; the human discerns.
5. **Per-stack deploy strategies** with automated rollback baked in.
6. **Visibility model:** daily/weekly digest, decision queue, incident-only synchronous pings.
7. **Calibration period:** first 2–4 weeks of operation, the digest includes a sample of auto-merged PRs for the human to spot-check.

This model trades a layer of safety (human-on-prod-deploy) for a layer of sustainability (gate that won't rot). It compensates by adding multiple AI safety layers and by intercepting the *irreversible* change categories at the only natural decision point (architectural review, framed as ADRs).

## Consequences

### Positive

- The human stays in the Wonder + Discernment lane (brainstorming, ADR review) and out of the execution lane.
- Velocity: PRs merge and ship without waiting on a human; deploys are 24/7.
- Irreversible-change risk is bounded by the ADR gate — those don't merge silently.
- Auto-rollback bounds the reversible-change risk.
- The pipeline does not require its operator to be alert and present to function safely.
- The platform itself becomes a strong portfolio piece: "I run my personal projects with AI shipping authority and an ADR-gated decision boundary" is a differentiating story.

### Negative

- **Aggressive trust assumption.** Requires the test suite, AI review battery, and health monitoring to actually catch what they claim to catch. If they don't, bad code ships.
- **Calibration cost.** First 2–4 weeks demand spot-checking discipline (mitigated by the digest sample mechanism). Skipping calibration converts the model into blind trust.
- **Detection coverage.** The destructive-change detector is rule-based; novel categories of irreversible change (e.g., a new "delete user" API endpoint) may not be caught. Mitigation: keep the rule set updated based on observed near-misses; periodically review the ADR-gated category list itself.
- **Onboarding friction for collaborators.** A future collaborator joining the project will see "AI auto-merges PRs" and may need calibration time too.

### Neutral

- The model is novel enough that it's worth documenting publicly (which we are, via this repo as a portfolio piece).
- The ADR-gated category list will likely be revised as we learn what it misses.

## Pros and Cons of the Options

### Sub-decision 1: Approval model

| Option | Trade-off |
|---|---|
| **Human approves every PR + every prod deploy** | Maximum safety on paper. In practice, rots under fatigue; the operator's working profile guarantees it won't be sustained. |
| **Human approves prod deploys only** | Velocity on dev/staging; gate on prod. Same rotting problem at the prod step. |
| **AI has full shipping authority; human gates only architectural decisions via ADRs** (chosen) | Aggressive but matches the operator profile. The forced-human-gate is reframed as decision-making (preferred role) rather than review (avoided role). |
| **Fully autonomous (no human gate anywhere)** | Maximum velocity; loses the irreversible-change safety net. |

### Sub-decision 2: Environment topology

| Option | Trade-off |
|---|---|
| **dev+prod only** | Simple; insufficient for live-data projects (no AI test bench between merge and users). |
| **dev+staging+prod** (chosen for live-data; configurable) | Staging gives testing agents a prod-like environment without touching prod. |
| **prod-only with feature flags** | Single env to manage; requires a flag platform, added complexity, not a starter pattern. |

### Sub-decision 3: Promotion mechanism

| Option | Trade-off |
|---|---|
| **Branch-per-environment** (`develop`, `staging`, `prod` branches) | Anti-pattern; produces drift between envs. Already excluded by ADR-0002. |
| **Trigger-based throughout** (every-merge → dev, then click-click-click) | Simple; no stable artifact identifier; harder to roll back ("which version was in prod?"). |
| **Tag-based for prod, trigger-based for dev/staging** (chosen) | Composes cleanly with Conventional Commits + release-please. Each tag is a stable artifact identified by a version. |
| **GitOps / pull-based** (ArgoCD, Flux) | Best practice for k8s. Massive overkill for solo non-k8s projects. |

### Sub-decision 4: Deploy strategy

Per-stack defaults rather than a single platform-wide choice: Lambda alias blue-green / ECS rolling / static atomic / VM blue-green. Projects can override via ADR. No real "decision" here beyond accepting per-stack defaults.

### Sub-decision 5: ADR-gated category list

| Option | Trade-off |
|---|---|
| **Just destructive DB migrations** | Catches the most-irreversible class; misses other irreversible/high-blast-radius categories. |
| **Five categories: destructive migrations, new external deps/services, security-relevant changes, API contract changes, schema changes** (chosen) | Covers the broader "irreversible or high-blast-radius" surface. Forces architectural reasoning at the right moments. |
| **Anything substantial** (subjective) | Subjective triggers rot — no clear detection rule means inconsistent application. |

- Standards doc: [`docs/standards/02-ci-cd.md`](../standards/02-ci-cd.md).
- Reusable workflows live under `workflows/` in this repo and are referenced from each scaffolded project. Implementation is Task #15.
